# Plan: Two-tier element selector menu for zellij-tab-config

## Goal

Port the two-tier group/field menu from `omarchy-theme-studio` to `zellij-tab-config`, replacing the current flat arrow-key-only navigation with a top-bar menu showing grouped element pills and field swatch chips.

## Context

**Current state (zellij-tab-config):**
- 14 `PreviewElement` variants (TabSelected, PaneSelected, TableTitle, etc.) — flat, no grouping
- Each element has both FG (`PreviewAttribute::Base`) and BG (`PreviewAttribute::Background`) attributes
- Navigation: arrow keys move through a flat vertical + horizontal order (up/down/left/right)
- Each element maps to a Zellij `ThemeComponentType` which has `.base` and `.background` colors
- The preview is a mock Zellij terminal layout (tab bar + split panes + status bar)

**Target (from omarchy-theme-studio):**
- 6 `PaletteGroup` variants → group pills at top (row 1)
- Each group's `PaletteField` variants → swatch chips with FG/BG preview (row 2)
- Navigation: ← → for groups, ↑ ↓ for fields within group, number keys for direct group jump
- Keyboard `/` for fuzzy field search

## Proposed approach

Add a two-tier grouping system to `PreviewElement` without changing the underlying Zellij theme model. Each element gets assigned to a `PreviewGroup`, and navigation is restructured around groups.

### Step 1: Add `PreviewGroup` enum

In `src/ui/state.rs`, add a new enum grouping the 14 `PreviewElement` variants:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub enum PreviewGroup {
    TabBar,     // TabSelected, TabUnselected1, TabUnselected2
    Panes,      // PaneSelected, TextSelected, PaneUnselected, PaneHighlight
    Tables,     // TableTitle, TableCellSelected, TableCellUnselected
    Lists,      // ListSelected, ListUnselected
    Status,     // ExitSuccess, ExitError, StatusBar
}
```

5 groups total. Each group gets `label()`, `fields()`, `index()`, `all()` methods.

Wire `PreviewElement` to its group via `fn group(&self) -> PreviewGroup`.

### Step 2: Replace flat navigation with group-aware navigation

Remove the existing `move_up/down/left/right` methods on `PreviewElement`. Instead, add group-level navigation:

- `App.selected_group: PreviewGroup` — the active group
- Within-group field selection: `selected` field stays as `PreviewElement` but always belongs to current group
- `prev_group()` / `next_group()` — ← → keys
- `prev_field_in_group()` / `next_field_in_group()` — ↑ ↓ keys (wraps within group)

Also: `selected_attribute` stays (FG vs BG), toggled with Tab.

### Step 3: Render the two-tier menu

In `src/ui/render.rs`, add two new methods (ported from omarchy-theme-studio):

1. **`render_group_tabs(frame, area)`** — Row 1: five group pills with active one highlighted
   - Uses chrome colors (consistent with the app's status bar style)
   - Shows number + label (e.g., `1 TabBar 2 Panes 3 Tables ...`)
   - Active group uses accent FG + accent BG

2. **`render_field_selector(frame, area)`** — Row 2: current group's fields as swatch chips
   - Each chip shows a small color swatch (FG-colored square next to BG-colored square) + field label
   - Selected field gets a highlighted border
   - This is the primary indicator of what element you're editing

Integrate these into `render_preview` — the vertical layout becomes:

```
[group tabs]        height 1
[field selector]    height 1
[tab bar]           height 1
[panes (Fill)]
[status bar]        height 1
```

### Step 4: Update event handling

In `src/ui/events.rs`:
- ← / → → `prev_group()` / `next_group()`
- ↑ / ↓ → `prev_field_in_group()` / `next_field_in_group()`
- `1`-`5` → jump to group by index
- `/` → open fuzzy field search (existing in omarchy-theme-studio, port as new feature)
- Tab → cycle FG/BG (unchanged)
- Remove the existing move_left/move_right/move_up/move_down paths in Preview mode

### Step 5: Update tests

`render.rs` has inline tests checking specific drawn coordinates. These will need to shift by +2 rows since the menu pushes everything down. Update expected Y coordinates in:
- `dracula_preview_buffer_carries_expected_theme_styles`
- `default_preview_right_pane_uses_unselected_frame_color`
- All other buffer-coordinate tests

## Files to change

| File | Change |
|---|---|
| `src/ui/state.rs` | Add `PreviewGroup` enum; add `selected_group` to `App`; replace element navigation with group navigation |
| `src/ui/render.rs` | Add `render_group_tabs()`, `render_field_selector()`; adjust `render_preview()` layout to reserve top 2 rows; update all buffer-coordinate tests |
| `src/ui/events.rs` | Remap Preview-mode keybindings to group-based navigation |

## Not changing

- Zellij `Theme` / `ThemeComponentType` model — these remain exactly as-is (each component still has `.base` and `.background`)
- KDL config parsing / exporting
- Color picker, theme loader, save/apply flows
- Any overlay rendering

## Risks

1. **Test breakage** — buffer coordinates in the inline render tests are position-sensitive. Every Y coordinate will need +2. Medium effort but mechanical.
2. **Navigation muscle memory** — current users navigate by "drilling" with arrow keys across a flat list. The new system requires ← → to switch groups first, then ↑ ↓ for fields. This is a UX change.
3. **Group boundaries** — some elements span visual regions (e.g. `PaneHighlight` is rendered in the right pane but conceptually a "Pane" element). The grouping should be by semantic category, not visual position.
4. **`PreviewAttribute::Base/Background` naming** — currently `Base` = FG, `Background` = BG. Could be confusing in the field selector context. Consider a clarifying rename but keep scope minimal.

## Open questions

- Are 5 groups the right split? Alternative: merge Tables+Lists into a single "Content" group (4 groups total).
- Should the field selector show both FG and BG for each element, or only the currently-selected attribute? Current proposal: show both as dual swatches.
- The omarchy-theme-studio has `/` fuzzy search — port this too or defer?
